# README 自動生成

プロジェクトの構造、コード、設定を深く解析し、**非エンジニアでもわかる**包括的な README.md（英語）と README_ja.md（日本語）を生成する。

**重要**: README は技術的な背景がない人でも理解できる内容にすること。専門用語は避けるか、必ず平易な言葉で説明を添える。

## 事前準備

1. **プロジェクトの深い理解**

   - プロジェクト全体の構造を把握
   - 主要な機能の実装箇所を特定
   - データフローとアーキテクチャを理解
   - 依存関係と技術スタックを分析
   - **推奨**: Serena MCP が利用可能な場合は使用すると効率的（必須ではない）

2. **既存ドキュメントの確認**
   - `dev-docs/` フォルダ内のドキュメントを確認
     - `dev-docs/commands/` に各コマンドの詳細ドキュメント（md）と処理フロー図（PlantUML）が存在する場合は、README からの参照リンクを追加
   - 既存の README があれば内容を比較してマージ
   - プロジェクト固有の情報を抽出

## 生成する README の構成

### README.md（英語版）

**非エンジニア向けの親しみやすい構成：**

#### 1. 📌 Name

- プロジェクト名
- 簡潔でわかりやすいキャッチフレーズ（1 行）
- 「このプロジェクトは〇〇をするためのツールです」という形式

#### 2. 📖 What is this?（これは何？）

**平易な言葉で説明：**

- このプロジェクトで「何ができるか」（機能ではなく、得られる価値）
- 「誰のために」作られたか
- 「どんな問題を解決するか」を具体的に
- 日常的なメタファーや例えを使って説明

**例：**
❌ 悪い例: "This is a REST API backend with PostgreSQL database"
✅ 良い例: "This tool helps you organize and track customer feedback, like having a digital notebook that automatically sorts comments"

**技術情報（折りたたみ可能）：**

- 使用している主要な技術（簡単な説明付き）
- 専門用語には必ず解説を添える
  - 例: "Node.js（サーバーを動かすためのプログラム実行環境）"

#### 3. 🔧 What You'll Need（必要なもの）

**平易な言葉で説明：**

- 必要なソフトウェア（「なぜ必要か」も説明）
  - 例: "Python 3.9 以上（このプログラムを実行するための言語。まだインストールしていない場合は[こちら](URL)からダウンロードできます）"
- 必要なアカウントやサービス（該当する場合）
- パソコンの動作環境（Windows/Mac/Linux）

**チェックリスト形式：**

```markdown
- [ ] Python 3.9 以上がインストールされている
- [ ] Git がインストールされている（コードをダウンロードするツール）
- [ ] テキストエディタ（メモ帳でも OK、VS Code 推奨）
```

#### 4. 🚀 Getting Started（始め方）

**超丁寧なステップバイステップ：**

各ステップで：

- 何をするか
- なぜそれが必要か
- 具体的なコマンド
- 成功したかどうかの確認方法
- よくあるエラーとその対処法

**例：**

```markdown
### Step 1: コードをダウンロード

まず、このプロジェクトのコードをあなたのパソコンにコピーします。

1. ターミナル（Mac）またはコマンドプロンプト（Windows）を開きます
2. 以下のコマンドをコピー＆ペーストして Enter を押します：

   `git clone https://github.com/...`

3. 成功すると「Cloning into...」というメッセージが表示されます

💡 **うまくいかない場合：** [トラブルシューティング](#troubleshooting)を確認してください
```

#### 5. ⚙️ Setup（セットアップ）

**設定ファイルの説明：**

- 環境変数とは何か（平易に説明）
- どの値を変更する必要があるか
- 表形式で各設定項目を説明

**表の例：**

| 設定項目     | 説明                                     | 例               |
| ------------ | ---------------------------------------- | ---------------- |
| DATABASE_URL | データの保存場所（変更不要の場合が多い） | `localhost:5432` |
| API_KEY      | 外部サービスとの接続に必要なパスワード   | `abc123...`      |

#### 6. 💻 How to Use（使い方）

**ユースケース中心の説明：**

- 「〇〇をしたい場合」という形式
- 実際の画面やコマンド実行例
- 期待される結果（スクリーンショットがあればベスト）

**CLI コマンドがある場合：**

- 基本的な使用例を記載
- より詳しい説明は `dev-docs/commands/` への参照を追加
- 各コマンドの詳細ドキュメント（説明 md ファイル）とフロー図（PlantUML）が存在することを明記

**例：**

```markdown
### プログラムを起動する

1. ターミナルで以下を実行：
   `npm start`

2. ブラウザで `http://localhost:3000` を開く

3. 以下のような画面が表示されれば成功です：
   [スクリーンショット or 説明]

💡 **より詳しく：** 各コマンドの詳細な使い方や処理の流れについては、[dev-docs/commands/](dev-docs/commands/) フォルダ内のドキュメントを参照してください。各コマンドの説明文書と処理フロー図が用意されています。
```

#### 7. 📁 Project Structure（フォルダ構成）

**各フォルダの役割を平易に説明：**

```text
project/
├── src/              # メインのプログラムファイル（ここを編集することが多い）
├── tests/            # プログラムが正しく動くかチェックするファイル
├── dev-docs/         # 詳しい技術ドキュメント
│   ├── commands/     # コマンドの詳細説明とフロー図（該当する場合）
│   └── ...           # その他の開発者向けドキュメント
└── config/           # 設定ファイル（基本的に触らない）
```

💡 **ヒント：** `dev-docs/commands/` フォルダには、各コマンドの詳しい使い方と処理の流れ図があります（該当する場合）

#### 9. 🆘 Troubleshooting（困ったときは）

**よくあるエラーと解決方法：**

```markdown
### エラー: "command not found"

**原因:** 必要なソフトウェアがインストールされていません
**解決方法:** [必要なもの](#what-youll-need)を確認してインストールしてください

### エラー: "Port 3000 is already in use"

**原因:** すでに別のプログラムが同じポートを使用しています
**解決方法:**

1. 他のターミナルウィンドウを確認し、既に起動していないか確認
2. または別のポート番号を使用（詳細は[こちら](#configuration)）
```

## 処理方針

1. **プロジェクトの深い解析**

   - プロジェクトの構造を体系的に理解
   - 主要な機能とその実装を特定
   - データフローとアーキテクチャを把握
   - **推奨**: Serena MCP が利用可能な場合は、より効率的かつ包括的な解析が可能（必須ではない）

2. **既存情報の活用**

   - 既存の README があれば内容を精査
   - 重要な情報は保持してマージ
   - 古い情報や不正確な情報は更新

3. **非エンジニア視点の徹底**

   - 技術的背景がない人を想定した表現
   - 専門用語は必ず平易な言葉で言い換えるか説明を追加
   - 「なぜそれが必要か」を必ず説明
   - 日常的なメタファーや例えを活用
   - チェックリスト、絵文字、視覚要素を積極的に使用

4. **実用性重視**

   - 新しいメンバーがすぐに始められる内容
   - よくある問題のトラブルシューティング（初心者が遭遇しやすいエラー）
   - 実際に動作するコード例（コピペで使える）
   - FAQ セクションで初心者の疑問を先回り

5. **メンテナンス性**
   - プロジェクトの成長に合わせて更新しやすい構造
   - 明確なセクション分け
   - 必要に応じて他ドキュメントへの参照
